# リポジトリ全体 コード品質改善方針

## 1. 目的

本書は、本リポジトリ全体に対して

- 命名
- コメント
- docstring
- 入力検証
- ファイル配置
- スクリプト整理
- PEP 8 準拠

をどう改善するかの基準を固定するための文書である。

本方針の対象は

- `src/`
- `examples/`
- `tests/`

にある Python ファイル全体である。

---

## 2. 現状の問題整理

今回の指摘は、単一ファイルではなく repo 全体の一貫性不足を指している。
主問題は以下である。

1. PEP 8 を意識した統一的な style rule が repo にない
2. module / class / function の役割を説明するコメントや docstring が不足している
3. 複雑な処理フローが helper の奥に沈み、どこで何をしているか追いにくい
4. 入力検証が処理本体と同じ粒度で並び、読み分けしにくい
5. 命名が API の使い方を自然に想起させない箇所がある
6. `examples/` と `tests/` の役割別整理方針が明文化されていない

---

## 3. repo-wide ルール

### 3.1 style

- Python コードは PEP 8 を基準とする
- import 順は `標準ライブラリ -> 外部ライブラリ -> 自モジュール` とする
- 1 行が長くなりすぎる場合は、式の意味が崩れない単位で改行する
- `pyproject.toml` に lint / format の基準を置く

### 3.2 コメントと言語

- module には module docstring を置く
- public class / public function には docstring を置く
- 独自用語や実装モードの違いは、日本語コメントで補う
- コメントは「何をしているか」だけでなく、「なぜその分岐や設計が必要か」を書く
- 単純代入や自明な 1 行にはコメントを書かない

### 3.3 関数構造

長い関数は少なくとも次の区分が見えるようにする。

1. 入力正規化
2. 入力検証
3. 状態更新
4. 本体処理
5. 出力整形

必要なら `# 入力検証`, `# 本体処理` のようなセクションコメントを入れる。

### 3.4 入力検証

- 同型の `if ...: raise ValueError(...)` が続く場合は helper 関数へ寄せる
- validation helper は「本体ロジックを隠す」のではなく、「検証の定型句を圧縮する」ためだけに使う
- 例外メッセージは、呼び出し側が何を直せばよいか分かる文にする

### 3.5 命名

- 単数と複数はインターフェースの粒度と一致させる
- `*_indices`, `*_count`, `*_counts_per_band` のように、返り値の shape が想像できる名前を優先する
- 既存 API を改名するときは、必要なら互換 alias を一定期間残す

### 3.6 処理フローの可視化

複雑な streaming / filterbank / beamforming 系では、
以下のように処理順が読める構造を優先する。

```python
frames = buffer.process(x_band)
for frame in frames:
    spectrum = fft(frame)
    statistics.update(spectrum)
    beamformed = beamformer.apply(spectrum)
    y_valid = ifft_and_extract(beamformed)
```

つまり

- 1 個の helper が何段も暗黙に処理を抱え込む構造
- validation と処理本体が視覚的に混ざる構造

は避ける。

### 3.7 `examples/` と `tests/` の整理方針

現在は次の階層を正式配置として採用し、`examples/` と `tests/` をこの構成へ再配置する。

- `examples/core/`
- `examples/filterbank/`
- `examples/beamforming/`
- `examples/nonuniform/`
- `tests/core/`
- `tests/filterbank/`
- `tests/beamforming/`
- `tests/nonuniform/`

再配置時は、既存文書の参照パスと example 内の `ROOT` 解決も合わせて更新する。

## 4. 今回の実装対象

今回の修正では、少なくとも以下を行う。

1. `pyproject.toml` に style / lint 基準を追加する
2. repo 全体へ module docstring を追加する
3. 複雑な中核実装に日本語 docstring / コメントを追加する
4. validation helper を追加し、重複した入力検証を整理する
5. 命名が分かりにくい API を、より説明的な名称へ整理する
6. 互換性が必要な箇所は alias を残す

---

## 5. 優先順位

1. `src/` の本体モジュール
2. `examples/` の実行スクリプト
3. `tests/` の補足コメントと整理

理由:

- 最初に読むべきなのはライブラリ本体である
- `examples/` は利用者が実行経路を学ぶ入口である
- `tests/` は仕様確認の補助線として最後に整える

---

## 6. 完了条件

今回の方針適用が完了したとみなす条件は以下である。

- repo 全体の Python ファイルに module docstring がある
- 主要 public class / function に説明がある
- 複雑モジュールではセクションコメントで処理の流れが読める
- validation helper により定型エラーチェックの見通しが改善している
- 変更後に全回帰試験が通る
