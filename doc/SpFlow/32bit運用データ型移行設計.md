# 32bit 運用データ型移行設計

## 1. 目的

実運用時のメモリ帯域、キャッシュ使用量、FFT・行列演算の処理時間を抑えるため、spflow の運用データを次の型へ統一する。

- サンプル数、チャネル番号、ビーム番号、周波数ビン番号: `int32`
- 実信号、座標、遅延、周波数軸、レベル系列: `float32`
- 複素スペクトル、ステアリング、ビーム重み、適応処理状態: `complex64`

Python の設定値や shape は Python `int` / `float` のまま扱う。これらは NumPy 配列の保存型ではなく、Python API と NumPy shape が要求するスカラー型だからである。

## 2. 運用経路の型契約

外部入力は各公開処理の入口で上記の運用型へ正規化する。FFT、STFT、FIR、Delay-and-Sum、MVDR、LCMV、GSC、SLC の公開出力も、特に明記しない限り `float32` または `complex64` とする。

配列添字は保存時に `int32` とする。一方、NumPy の shape、slice、`range`、単一要素アクセスへ渡す直前では Python `int` へ変換する。

## 3. 64bit を許可する境界

次の処理は、32bit の丸め誤差が採否や安定性を変える場合に限り、局所的な `float64` / `complex128` 計算を許可する。

- FIR・フィルタバンク係数のオフライン設計
- 悪条件な共分散行列の分解、条件数評価、設計時の連立方程式
- 人間向け評価指標の集計と dB 変換
- JSON/CSVへ保存する設計値の生成

64bit に昇格する箇所には理由をコメントし、運用経路へ戻す境界で `float32` / `complex64` へ明示的に変換する。単に NumPy の既定型だからという理由では64bitを残さない。

## 4. 評価方針

32bit化の採否は実行成功だけで判断しない。変更前後で以下を確認する。

- 狭帯域・広帯域の入力帯域積分 RMS
- Delay-and-Sum のピーク方位誤差、主ローブレベル、第一副極
- noise-only の `w^H R_n w` と `10log10(N)` アレイゲイン
- target+noise の成分別電力整合
- MVDR/SLC の制約応答、共分散条件数、波形連続性
- NaN/inf、出力 shape、公開出力 dtype

許容差は `float32` / `complex64` の演算順による誤差と、信号処理上検出したい異常を分離して設定する。許容差を変更する場合は観測最大誤差と検出目的をテストコメントへ記録する。

## 5. Pyright 完了条件

`src`、`tests`、`evaluations`、`examples` を含むリポジトリ全体で Pyright error / warning が0件であることを完了条件とする。`Any`、`cast`、設定による診断抑制でエラーを隠さず、Optional分岐、固定された戻り値型、具体的なNumPy配列型によって解消する。
